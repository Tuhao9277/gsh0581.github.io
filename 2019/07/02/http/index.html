TCP协议的三次握手
发表于 2019-07-02| 更新于 2019-07-02

三次握手连接的建立

  • 客户端与服务器进行http发生返回过程中，需要创建一个TCP connection
  • HTTP只有请求和响应的概念，不存在连接。
  • TCP里创建一个二者的通道。可存在多个http请求，可以通过声明让其一直保持tcp连接

包含两个序号 三个标志位

序号

  • seq 序号 用来表示向服务器发送的字节流
  • ack 确认序号 ACK =1时，ack才有效，ack = seq+1；

标志位

  • ACK：确认信号有效
  • SYN：发起一个新连接
  • FIN：释放一个新连接

client与server的三次握手过程

  • 第一次发送：客户端发起SYN=1，Seq =x;
  • 第二次：服务器端接收并发送 SYN =1,ACK =1 ，seq = y,ack = x+1; 使用了ack对Ade数据包进行确认
    服务器收到了seq = x的数据包，准备接收seq+1的数据包，于是ack =x+1，服务器告诉客户端自己的数据包为y
  • 第三次：客户端接收并发送 ACK = 1，ack = y+1; seq = x+1；，服务端收到ack = seq+1，连接建立

##为什么要进行第三次握手？为什么A还要发一次确认？

当某一建立连接的请求报文在发送过程中滞留，以至于连接释放时还没过去，导致失效
但服务器收到该请求时误以为客户端又有新的连接请求，就向客户端发确认，由于现在client并没有发出建立连接的请求，
因此不会理睬server的确认，也不会向server发送数据。在没有三次握手的情况下，server以为新的运输连接已经建立，
并一直等待client发来数据。这样，server的很多资源就白白浪费掉了。
